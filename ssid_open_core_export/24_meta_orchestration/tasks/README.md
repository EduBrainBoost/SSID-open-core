# TaskSpec System

## Overview

TaskSpecs are YAML files in `24_meta_orchestration/tasks/specs/`.
Each spec defines a scoped, auditable unit of work for agent-driven development.

## Getting Started

Follow these steps to create and execute a TaskSpec:

### Step 1: Create the TaskSpec YAML

Create a new `.yaml` file under `24_meta_orchestration/tasks/specs/`.
Use the naming convention `<TASK_ID>.yaml` (e.g., `POC_TOOLING_REF_001.yaml`).

### Step 2: Define all required fields

See the Required Fields table below. Every field must be present.

### Step 3: Validate the spec

```bash
python 12_tooling/cli/automation_loop.py --verify-spec --spec 24_meta_orchestration/tasks/specs/YOUR_TASK.yaml
```

Fix any validation errors before proceeding.

### Step 4: Create a feature branch

```bash
git checkout -b task/<TASK_ID>
```

### Step 5: Initialize and start the agent run

```bash
python 12_tooling/cli/automation_loop.py --init
python 12_tooling/cli/automation_loop.py --start --task <TASK_ID> --spec 24_meta_orchestration/tasks/specs/YOUR_TASK.yaml
```

### Step 6: Implement changes

Work within the `scope_allowlist` paths. Do not touch `forbidden_paths`.

### Step 7: Commit, finalize, and push

```bash
git add <your-files> && git commit -m "description of changes"
python 12_tooling/cli/automation_loop.py --finalize --task <TASK_ID>
git push -u origin task/<TASK_ID>
```

### Step 8: Open a PR

All stability gates must pass before merge is allowed.

## Example TaskSpec

Below is a complete example TaskSpec YAML with all required fields:

```yaml
task_id: "POC_TOOLING_REF_001"
title: "Refactor CLI tooling entry points"
task_type: "tooling"

scope_allowlist:
  - "12_tooling/cli/"
  - "12_tooling/cli/docs/"

forbidden_paths:
  - ".github/"
  - "01_ai_layer/"

required_checks:
  - "stability_gate"
  - "sot_validator"
  - "pytest"

acceptance_criteria:
  - "All CLI entry points use consistent argument parsing"
  - "No root-level files created or modified"
  - "All existing tests pass"

evidence_outputs:
  - "02_audit_logging/agent_runs/POC_TOOLING_REF_001/evidence.json"
  - "02_audit_logging/agent_runs/POC_TOOLING_REF_001/patch.diff"

notes: "Part of the bootstrap phase. Focus on consistency, not new features."
```

## Required Fields

| Field | Type | Description |
|-------|------|-------------|
| `task_id` | string | Unique identifier (e.g., `POC_TOOLING_REF_001`) |
| `title` | string | Human-readable title |
| `task_type` | enum | `bootstrap`, `tooling`, `tests`, `docs` |
| `scope_allowlist` | list | Paths the task may modify |
| `forbidden_paths` | list | Paths that must never be touched |
| `required_checks` | list | Ordered checks to run (read-only) |
| `acceptance_criteria` | list | Concrete, testable criteria |
| `evidence_outputs` | list | Expected repo paths for evidence |
| `notes` | string | Optional context |

## Workflow

1. Author creates a TaskSpec YAML under `specs/`.
2. `automation_loop.py --verify-spec --spec <path>` validates the spec.
3. `automation_loop.py --start --task <TASK_ID> --spec <path>` initializes an agent run.
4. Agent implements changes within `scope_allowlist`.
5. `automation_loop.py --finalize --task <TASK_ID>` runs checks and produces evidence.
6. Human reviews PR. Merge requires all gates PASS.

## Common Patterns and Tips

### Scoping: narrow is better

Keep `scope_allowlist` as narrow as possible. A task that touches
`12_tooling/cli/` and `12_tooling/cli/docs/` is better than one that allows
all of `12_tooling/`. Narrow scopes make auditing and review faster.

### One task, one concern

Each TaskSpec should address a single concern (one feature, one fix, one
set of related docs). If you find yourself listing many unrelated paths in
`scope_allowlist`, consider splitting into multiple TaskSpecs.

### Forbidden paths as guardrails

Use `forbidden_paths` to protect areas that are definitively out of scope.
Common entries include `.github/`, other numbered root directories you should
not touch, and configuration files at repo root.

### Naming conventions

- Task IDs follow the pattern: `<PHASE>_<AREA>_<TYPE>_<SEQ>`
  - Example: `POC_TOOLING_REF_001` (POC phase, tooling area, refactor type, sequence 001)
- Keep IDs uppercase with underscores
- Sequence numbers allow multiple related tasks in the same area

### Evidence outputs

Always include at least `evidence.json` and `patch.diff` in `evidence_outputs`.
These are generated by `--finalize` and are required for audit compliance.

### Testing before finalize

Run the stability gate locally before finalizing:

```bash
python 12_tooling/cli/stability_gate.py --run
```

This catches issues early and avoids failed CI checks after push.

## Rules

- No secrets in specs.
- No root-level writes.
- PR-only workflow; no direct-to-main merges.
- All evidence is hash-only (no raw logs persisted in repo).
