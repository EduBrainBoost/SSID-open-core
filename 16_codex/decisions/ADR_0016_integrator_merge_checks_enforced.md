# ADR-0016: Integrator Merge Checks Enforced via CI

## Status
Accepted (amended 2026-02-28 — Scope Allowlist Guard added)

## Context
Until now, Integrator-Merge-Checks (scope validation, ROOT-24-LOCK, ADR-Pflicht,
forbidden metadata) were applied manually before each merge. Manual enforcement is
error-prone and non-deterministic. The repo_separation_guard.py already catches
some violations during the Gate Chain, but coverage gaps remain — particularly for
forbidden metadata files and an explicit PR-level summary.

Additionally, there was no CI-enforced mechanism to restrict PR changes to
authorized zones within the repo structure. The ROOT-24-LOCK guard prevents new
root entries but does not validate that changes stay within declared scope zones.

## Decision
Add a dedicated GitHub Actions workflow `integrator_merge_checks.yml` that runs
on every `pull_request` event (opened, synchronize, reopened, ready_for_review)
and blocks merge on failure. The workflow enforces four automated checks:

1. **Scope Allowlist Guard** — Reads authorized path prefixes from
   `.github/scope_allowlist.txt` (plaintext, one prefix per line, `#` comments).
   Hard-fail: if allowlist is missing or empty, the guard FAILs — no escape
   hatches. Prefix matching via fixed-string comparison (`[[ "$f" == "$p"* ]]`),
   no regex, no YAML parsing. The default allowlist includes all ROOT-24
   directories, `.github/`, and allowed root files per
   `23_compliance/exceptions/root_level_exceptions.yaml`.

2. **No-Forbidden-Metadata Guard** — Fails if the PR introduces files matching
   forbidden patterns: `.claude/`, `.devcontainer/`, `.ssid_sandbox/`,
   `*.code-workspace`, `.DS_Store`, `Thumbs.db`, `.env`/`.env.*`, `secrets/`.

3. **ROOT-24-LOCK Guard** — Compares root-level directory/file entries in the PR
   against the base branch. Fails if any new root-level entry is introduced that
   does not exist on `main`. This is a dynamic check (no hardcoded list).

4. **ADR-Pflicht Guard** — Mirrors `repo_separation_guard.py` ADR_TRIGGER_PREFIXES:
   changes in `.github/workflows/`, `16_codex/governance/`, `16_codex/agents/`,
   or `24_meta_orchestration/` require an `05_documentation/adr/ADR_*.md` in the
   same PR. Fails if trigger zone is hit without an accompanying ADR.

Additionally:
- `.github/scope_allowlist.txt` serves as the authoritative config for allowed
  change zones. Changes to this file itself require ADR.
- CODEOWNERS requires review for scope allowlist changes.
- The workflow is added as a **required status check** in branch protection,
  alongside the existing Gate Chain, Gitleaks, and Evidence Verification checks.

## Allowlist Format

```
# .github/scope_allowlist.txt
# 1 prefix per line. Lines starting with # are comments. Blank lines ignored.
.github/
01_ai_layer/
...
24_meta_orchestration/
README.md
```

Semantics: a changed file passes if its path starts with any listed prefix.
No regex, no glob expansion. Pure fixed-string prefix match.

## Trigger Rules (ADR-Pflicht)

| Prefix | ADR Required |
|--------|-------------|
| `.github/workflows/` | Yes |
| `16_codex/governance/` | Yes |
| `16_codex/agents/` | Yes |
| `24_meta_orchestration/` | Yes |
| All other paths | No |

## Failure Semantics

| Condition | Result |
|-----------|--------|
| `.github/scope_allowlist.txt` missing | FAIL |
| `.github/scope_allowlist.txt` empty (no prefixes) | FAIL |
| Any changed file outside all prefixes | FAIL |
| All changed files within at least one prefix | PASS |

## Consequences
- Every PR is automatically validated before merge. No manual enforcement needed.
- PRs that introduce forbidden files, violate ROOT-24-LOCK, touch unauthorized
  scope zones, or omit required ADRs are blocked at the CI level.
- The workflow adds ~10s to PR check time (pure `git diff` + bash, no dependencies).
- New ADR-trigger prefixes must be added in both `repo_separation_guard.py` and
  `integrator_merge_checks.yml` to maintain consistency.
- New scope zones must be added to `.github/scope_allowlist.txt`.
- Branch protection must list `Integrator Merge Checks` as a required status check.
- No WORM/Evidence artifacts are generated by this workflow. Evidence is
  audit-logged separately via the Gate Chain and dispatcher.
